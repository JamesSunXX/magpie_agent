# Loop 自动提 MR 设计

## 背景

当前 `magpie` 的 `loop` 已经具备这些能力：

- 自动建分支
- 阶段完成后自动提交
- 全部阶段完成后给出完成结果

但它还缺最后一步：当开发和验证都完成后，自动把当前分支推上去并创建一个 GitLab MR。

现在如果想提 MR，只能在 `loop` 跑完后再手工做。这个缺口会带来两个问题：

1. 自动化只做到“代码完成”，没有做到“进入评审入口”。
2. 如果 MR 创建失败，系统也没有统一记录和通知，容易让人误以为整个流程都完成了。

本次只补 `loop` 完成后的 MR 收尾动作，不改 `harness` 的外层职责，不做每阶段单独提 MR。

## 目标

让 `loop` 在满足下面条件时自动尝试创建一个 GitLab MR：

1. 整个 `loop` 已经成功完成。
2. 开发和验证已经通过。
3. 配置显式开启了 MR 功能。
4. 当前不是 `dry-run`，并且存在可用分支。

同时满足下面的失败语义：

1. MR 创建成功：记录结果、保存链接、发通知。
2. MR 创建失败：`loop` 仍然算开发完成，但必须明确记录“需要人工补做 MR”，并发通知说明原因。

## 非目标

本次明确不做：

- `harness` 完成后再统一提 MR
- 每个阶段单独提一个 MR
- GitHub 或其他代码平台支持
- 自动合并、自动指派、自动加 reviewer
- 失败后自动重试多次创建 MR
- 用 MR 结果反向修改 `loop` 的成功/失败判定

## 方案

采用“`loop` 完成后的可选收尾动作”方案。

`loop` 的主流程保持不变：

1. 建分支
2. 执行开发与验证
3. 阶段内自动提交
4. 全部阶段完成后，标记 `loop` 完成

只有在第 4 步之后，才根据配置决定是否追加一次“推分支并创建 MR”。

这样做的原因是：

- MR 属于交付收尾动作，不属于阶段执行本身
- 现有自动建分支、自动提交都在 `loop` 内，MR 放在同一层最顺
- 即使 MR 失败，也不应该推翻“开发已经完成”这个事实

## 配置

在 `capabilities.loop` 下新增 MR 配置，默认关闭：

```yaml
capabilities:
  loop:
    mr:
      enabled: true
```

第一版只保留最小开关，不额外引入多余选项。

理由是：

- 当前目标是先打通自动提 MR 的主链路
- 目标分支、模板、创建方式优先复用仓库现有约定
- 先避免把配置做得过深，增加维护成本

## 执行顺序

当 `loop` 完成时，执行顺序调整为：

1. 将 `loop` 本身标记为开发完成
2. 如果 `mr.enabled` 未开启，直接按当前行为结束
3. 如果 `mr.enabled` 开启，检查是否满足提 MR 前提
4. 满足前提时，尝试推送当前分支并创建 1 个 GitLab MR
5. 将 MR 结果写入会话产物
6. 发出成功或失败通知
7. 返回最终 `loop` 结果

MR 行为只运行一次，不参与阶段重试逻辑。

## 前提条件

只有同时满足下面条件，才会尝试提 MR：

- `loop` 总体成功
- `mr.enabled = true`
- 不是 `dry-run`
- 当前会话有明确分支名
- 当前会话还没有成功提过 MR

如果不满足这些前提，不视为错误，只记为“本次未触发自动提 MR”。

## 创建方式

优先复用仓库现有的 GitLab MR 约定。

第一优先级：

- 如果仓库里有现成的 MR 创建脚本或约定入口，直接调用它

第二优先级：

- 使用通用的 Git 推送并创建 MR 的方式

这样做的原则是：

- 优先尊重仓库已有工作流
- 没有仓库级脚本时，也能保证最小可用

本次只要求“能创建 MR 并拿到可回显的结果”，不要求复杂模板渲染或二次同步。

## 结果落盘

新增一份独立的 MR 结果产物，记录：

- 是否启用了自动提 MR
- 是否实际执行了 MR 动作
- 分支名
- 创建是否成功
- MR 链接（成功时）
- 失败原因（失败时）
- 是否需要人工补做

这样做是为了把“开发完成结果”和“MR 收尾结果”明确拆开，避免后续查看状态时混淆。

同时，会话摘要也要能直接表达这层语义：

- 开发完成，MR 已创建
- 开发完成，但 MR 需要人工补做

## 通知

MR 结果必须发通知。

### 成功通知

通知内容至少包含：

- 开发已完成
- MR 已创建
- 分支名
- MR 链接

### 失败通知

通知内容至少包含：

- 开发已完成
- MR 未创建成功
- 需要人工补做
- 失败原因

这里的重点是通知文案不能把 MR 失败说成开发失败。

## 失败语义

MR 创建失败时，`loop` 的最终状态仍然保持完成，不改成失败。

原因是：

- 代码开发和验证已经完成
- MR 只是收尾交付动作
- 用户要求失败时由人工补做，而不是推翻整个开发结果

但失败不能静默处理，必须同时做到：

1. 写入会话产物
2. 更新最终摘要
3. 发出清晰通知

## 边界情况

需要明确处理这些情况：

1. `dry-run`
   - 不提 MR

2. 没有分支名
   - 不提 MR
   - 记为需要人工补做

3. `loop` 失败或暂停
   - 不提 MR

4. 同一会话已成功提过 MR
   - 不重复创建

5. 推送成功但 MR 创建失败
   - 仍算“MR 需要人工补做”
   - 保留已知信息，方便人工续做

6. 创建动作返回了非结构化输出
   - 只要能识别成功链接，就按成功处理
   - 否则按失败处理并保留原始信息

## 改动范围

### 代码改动

主要改动点：

1. `src/platform/config/types.ts`
   - 新增 `capabilities.loop.mr` 配置结构

2. `src/platform/config/init.ts`
   - 在初始化配置里补出默认项

3. `src/capabilities/loop/application/execute.ts`
   - 在 `loop` 成功结束后接入 MR 收尾动作
   - 写入 MR 结果产物
   - 调整最终摘要

4. `src/platform/integrations/notifications/`
   - 复用现有通知路由，补 MR 成功/失败消息发送

5. 文档
   - 更新 `README.md`
   - 更新 `docs/references/capabilities.md`

### 测试改动

至少覆盖：

1. 开关关闭时，行为与现在完全一致
2. 开关开启且创建成功时，能记录 MR 链接
3. 开关开启但创建失败时，`loop` 仍然完成
4. 失败时会留下“需要人工补做 MR”的记录
5. `dry-run` 不触发 MR
6. 没有分支时不触发 MR
7. 已经提过 MR 时不重复创建

## 风险与控制

### 风险

1. 不同仓库的 GitLab MR 创建方式不一致
2. 创建结果可能是非结构化文本，难以统一解析
3. 用户可能把“MR 失败通知”误读成“开发失败”

### 控制

1. 先走仓库已有约定，没有再走通用方式
2. 成功判定尽量只依赖最稳定的结果，例如可识别链接
3. 通知和摘要里强制把“开发完成”和“MR 待补做”分开表述

## 验收标准

满足下面结果即可认为本次能力完成：

1. 配置关闭时，`loop` 行为不变
2. 配置开启时，`loop` 成功完成后会自动尝试提 1 个 MR
3. MR 成功时，会话结果里能看到链接，通知里也能看到链接
4. MR 失败时，`loop` 仍显示开发完成，但会明确提示需要人工补做
5. `dry-run`、失败会话、暂停会话不会误提 MR
