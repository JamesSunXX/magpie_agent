# Stage AI Feishu Notifications Design

## Summary

Magpie will gain a built-in stage-level notification capability for both `loop` and `harness`. Every stage transition will be eligible for notification, including repeated entries into the same stage, and the notification body will be generated by an AI summarizer instead of a fixed static template.

The new behavior applies to all repositories that run Magpie once the feature is enabled in config. Notifications will include session information, stage information, a short AI-written summary, the list of AI tools/models involved in the stage, and a per-AI responsibility summary. Feishu is the first supported delivery target for stage summaries.

## Goals

- Send Feishu notifications for every `loop` and `harness` stage transition.
- Cover `entered`, `completed`, `failed`, `paused`, and `resumed` events.
- Send notifications even when the same stage is entered multiple times.
- Use AI to summarize stage context before dispatching the message.
- Include session identity, stage identity, AI roster, and AI responsibilities in every message.
- Keep the feature global to Magpie rather than per-repository custom glue.
- Ensure notification failures never block `loop` or `harness`.

## Non-Goals

- Reworking the existing completion/failure human-confirmation notification flow outside stage-aware additions.
- Building a generic async job queue for every provider in this iteration.
- Supporting rich stage-aware summaries for macOS or iMessage in this iteration.
- Replacing existing event logs or knowledge summaries.
- Guaranteeing perfect per-model action provenance when the runtime does not expose it.

## Current Constraints

- `loop` already emits lifecycle events to `events.jsonl` and dispatches notifications only for `loop_completed`, `loop_failed`, `loop_paused`, and human confirmation.
- `harness` emits stage events to its own event log but does not dispatch stage-aware notifications.
- Notification routing currently understands only a small fixed set of event types.
- Feishu payload generation is currently static and based on title + message text.
- Capability runtime already has enough context to know stage name, session id, high-level summary, and provider bindings, but not a single unified “stage notification context” object.

## Proposed Architecture

### 1. Stage notification domain model

Introduce a stage-aware notification layer that sits above providers and below capabilities.

It will define:

- stage event kinds for `entered`, `completed`, `failed`, `paused`, `resumed`
- capability scope for `loop` and `harness`
- a normalized stage context payload
- AI roster metadata for the current stage
- an occurrence counter for repeated entry into the same stage

This model becomes the single input to AI summarization and downstream notification delivery.

### 2. AI-generated message builder

Add a summarization component that accepts normalized stage context and returns:

- notification title
- notification body
- structured AI list
- structured AI responsibility summary

The summarizer should support two modes:

- normal mode: call a configured model/tool to produce the message
- fallback mode: emit a deterministic plain-text message when summarization fails

This keeps the feature resilient without downgrading the user experience to fully static templates on the happy path.

### 3. Capability integration points

#### Loop

`loop` will emit and notify at these points:

- when a stage starts
- when a stage finishes successfully
- when a stage fails
- when the run pauses for human review
- when the run resumes after pause

Repeated entry into the same stage must increment an occurrence count and still dispatch.

#### Harness

`harness` will emit and notify at these points:

- when outer stages change (`queued`, `developing`, `reviewing`, `completed`, `failed`)
- when review cycles complete
- when outer workflow fails
- when outer workflow completes

This gives users both the outer workflow view and the inner `loop` stage view.

### 4. Notification routing

Extend the notification router to support new stage-aware event types while keeping existing routes valid.

Routing rules remain config-driven:

- users can enable or disable stage-aware notifications globally
- users can route stage-aware events to `feishu_team`
- existing `loop_completed` / `loop_failed` routes continue to work

### 5. Feishu delivery format

Feishu remains a delivery provider only. It should not know how to summarize stage context.

It will accept the already summarized title/body and render them into Feishu `text` or `post` format.

The Feishu message should consistently show:

- session title
- capability + stage + event kind
- occurrence count when relevant
- AI-generated summary
- AI roster
- AI responsibilities
- next action or blocker

## AI Metadata Design

Every stage message needs two related but different outputs:

1. **AI roster**
   - the set of tools/models/agents materially involved in the stage
   - examples: `codex`, `kiro:architect`, `gemini`

2. **AI responsibilities**
   - one short sentence per AI/tool explaining its role in the stage
   - examples:
     - `codex`: main execution and code changes
     - `kiro:architect`: structure and risk review
     - `gemini`: counter-argument and edge-case challenge

### Data collection rules

- Prefer actual runtime bindings when available.
- For `loop`, collect planner/executor bindings from loop runtime config and known stage behavior.
- For `harness`, collect active reviewer ids, routing choices, and loop binding artifacts.
- When exact provenance is unavailable, summarize only what is defensible from configuration and the current execution path.
- Never invent detailed per-AI activity beyond what context supports.

## Configuration Design

Add a new optional block under notifications integration, for example:

```yaml
integrations:
  notifications:
    enabled: true
    stage_ai:
      enabled: true
      provider: codex
      max_summary_chars: 900
      include_loop: true
      include_harness: true
      events:
        - stage_entered
        - stage_completed
        - stage_failed
        - stage_paused
        - stage_resumed
    routes:
      stage_entered:
        - feishu_team
      stage_completed:
        - feishu_team
      stage_failed:
        - feishu_team
      stage_paused:
        - feishu_team
      stage_resumed:
        - feishu_team
```

### Configuration rules

- Stage-aware notifications are disabled unless `stage_ai.enabled` is true.
- If stage-aware notifications are enabled but summarizer config is incomplete, Magpie should use fallback summaries instead of failing the run.
- Existing notification config remains backward-compatible.

## Failure Handling

- If stage summary generation fails, fall back to a deterministic plain-text summary.
- If Feishu dispatch fails, record the failure in logs/results but do not fail `loop` or `harness`.
- If AI metadata cannot be fully resolved, emit partial AI context rather than blocking notification delivery.
- If stage event recording succeeds but notification dispatch fails, the stage still proceeds normally.

## Testing Strategy

### Unit tests

- stage event normalization
- repeated-entry occurrence counting
- AI roster extraction for `loop`
- AI roster extraction for `harness`
- fallback summary generation
- Feishu payload rendering of AI-rich content

### Capability tests

- `loop` dispatches on stage entry/completion/failure/pause/resume
- repeated stage entry emits multiple messages with incremented occurrence count
- `harness` dispatches on outer stage transitions and review cycle milestones
- notification failure does not fail the workflow

### Config tests

- new stage-aware config loads correctly
- old configs remain valid
- routes for new notification event types validate correctly

## Implementation Boundaries

Expected code areas:

- `src/platform/config/types.ts`
- `src/platform/integrations/notifications/types.ts`
- `src/platform/integrations/notifications/factory.ts`
- `src/platform/integrations/notifications/providers/feishu-webhook.ts`
- new stage-summary helpers under `src/platform/integrations/notifications/`
- `src/capabilities/loop/application/execute.ts`
- `src/capabilities/workflows/harness/application/execute.ts`
- matching tests under `tests/platform/notifications/`, `tests/capabilities/loop/`, and `tests/capabilities/workflows/`

## Open Decisions Resolved In This Design

- Repeated stage entry sends repeated notifications: yes.
- `entered`, `completed`, `failed`, `paused`, and `resumed` all notify: yes.
- Session info must appear in title: yes.
- AI roster and AI responsibility summaries must both appear: yes.
- This is a Magpie-wide built-in capability, not repository-local glue: yes.
