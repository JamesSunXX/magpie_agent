# Loop 自动提交 AI Commit Message 设计

## 背景

当前 `magpie` 的 loop 自动提交在每个阶段完成后直接使用固定文案：

- `feat(loop): 完成<stage>`

这个行为有两个问题：

1. 提交信息不反映真实改动内容，只反映阶段名。
2. 提交信息不遵循当前项目约定的中文规范和约定式格式要求。

本次只改 `magpie` 的自动提交逻辑，不改人工提交时的行为。

## 目标

让 loop 自动提交在有改动时：

1. 先基于本次暂存改动生成提交摘要。
2. 调用 AI 生成中文提交信息。
3. 提交信息符合既定规则：
   - 中文
   - 简洁
   - 祈使句
   - 约定式提交格式
4. 如果 AI 生成失败或结果不合法，自动退回默认提交信息，不能阻塞 loop。

## 方案

采用“单独抽一个自动提交信息生成器”的方案。

### 结构

在 loop 执行路径里保留原有自动提交时机不变，只替换“提交信息来源”：

1. 检查当前工作区是否有改动。
2. `git add -A`
3. 读取已暂存改动摘要
4. 调用 AI 生成提交信息
5. 校验输出
6. 执行 `git commit`

### 新增能力

新增一个独立的自动提交信息生成器，职责只有两件事：

1. 从当前仓库提取足够短的改动摘要
2. 基于固定 system prompt 生成提交信息

它不负责决定是否提交，也不负责 git 操作本身。

### 模型来源

自动提交信息生成器复用 loop 当前执行使用的模型绑定。

这样可以保持：

- 行为一致
- 不额外引入新的配置项
- 不要求用户再单独配置 commit 专用模型

### Prompt 规则

system prompt 固定使用用户提供的中文提交规范，要求输出：

- 首选单行标题
- 必要时允许正文
- 标题必须满足约定式提交格式

用户输入部分只包含：

- 当前阶段名
- 仓库中的已暂存改动摘要

不传完整大 diff，避免生成过慢和噪音过大。

## 改动范围

### 代码改动

主要改动点：

1. `src/capabilities/loop/application/execute.ts`
   - 把当前硬编码提交文案逻辑改为调用生成器

2. 新增自动提交信息生成器模块
   - 负责摘要提取
   - 负责 AI 调用
   - 负责结果校验和兜底

### 不改内容

本次明确不改：

- 手工提交逻辑
- 其他 workflow 的提交逻辑
- 配置文件结构
- loop 的自动提交开关与触发时机

## 失败兜底

如果出现任一情况：

- AI 调用失败
- 返回空内容
- 返回内容不满足约定式提交基本格式

则直接退回默认文案：

- `feat(loop): 完成<stage>`

这样保证自动提交最多“信息不够好”，不会“直接不提交”。

## 校验规则

对 AI 结果只做轻量校验：

1. 非空
2. 首行长度合理
3. 首行匹配 `<type>(<scope>):<描述>` 或 `<type>:<描述>`

如果不满足，就走兜底。

## 测试

至少覆盖下面几类：

1. 生成成功时使用 AI 文案提交
2. AI 返回空内容时回退默认文案
3. AI 返回非法格式时回退默认文案
4. 无改动时保持不提交
5. 分支不匹配时保持原有保护逻辑

## 风险与控制

### 风险

1. AI 生成结果不稳定
2. 提交摘要过长导致生成变慢
3. 新逻辑影响原有自动提交稳定性

### 控制

1. 只用短摘要，不喂完整 diff
2. 任何异常都回退默认文案
3. 保持 git 提交流程主体不变，只替换 message 来源
