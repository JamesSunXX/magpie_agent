# Harness Project Document Mode

## 背景

当前 `harness` / `loop` 在前置阶段会直接生成一批文档产物，但它默认按通用习惯落盘，不理解不同项目自己的文档规则。

这会带来两个直接问题：

- 正式文档和过程产物混在一起，容易把 `docs/` 根目录写乱
- 即使项目已经在 `AGENTS.md`、工作流文档或模板里写了清晰规则，`harness` 也不会优先遵守

对文档规则严格的仓库来说，这不是展示问题，而是流程不匹配问题。

## 目标

- 正式文档落点优先由模型结合项目自己的规则来判断，而不是在代码里写死
- 过程产物统一写到 `.magpie/sessions/...`，不再直接进入正式文档区
- 当模型无法可靠判断正式文档落点时，退回到保守默认位置，而不是污染项目 `docs/`
- 同一会话内的文档落点判断保持一致，避免不同阶段各写各的

## 非目标

- 不迁移项目里已经存在的旧文档
- 不要求所有项目都采用同一套固定目录结构
- 不在第一版强制校验 PRD / Story / TRD 的审批状态
- 不让模型自由决定“过程产物是否也可以进入正式文档区”

## 核心原则

### 1. 正式文档落点由项目规则驱动

系统不在 `harness` 或 `loop` 中硬编码“正式文档一定写到 `docs/[version]/[feature-name]/...`”。

相反，系统会在每次会话开始时收集以下上下文，交给模型判断正式文档的目标落点：

- 项目根目录 `AGENTS.md`
- 项目内工作流和文档规则文件
- 当前仓库 `docs/` 结构
- 文档模板目录

模型输出的是一份结构化“文档计划”，而不是直接生成正文。

### 2. 过程产物永远不进入正式文档区

无论模型如何判断，以下这类内容都只能视为过程产物：

- 阶段分析
- 临时检查稿
- 阶段报告
- 中间比对稿
- 阶段证据
- 仅服务于本轮执行的草稿和摘要

它们统一进入 `.magpie/sessions/...`，不允许直接写到项目正式 `docs/`。

### 3. 同一会话只做一次文档落点判断

每次 `harness` / `loop` 会话开始时做一次文档计划，后续各阶段复用。

这样可以避免：

- 同一轮内不同阶段把正式文档写到不同位置
- 每生成一个文件都重新问一次模型
- 会话恢复后重新猜一遍落点

## 设计

### A. 新增“文档计划”阶段

在 `loop run` 与 `harness submit` 的正式执行前，新增一段轻量文档规划动作。

输入：

- `AGENTS.md`
- 项目文档规则文件
- 当前 `docs/` 目录树
- `docs/templates/` 中可用模板
- 当前任务目标、PRD 路径、阶段列表

输出：

- 这次会话是否启用项目文档模式
- 正式文档根位置
- 每类正式文档的目标路径
- 哪些产物是过程产物
- 使用的规则依据
- 模型对该判断的把握程度
- 是否触发保守回退

建议将该计划持久化到：

- `.magpie/sessions/<capability>/<session-id>/document-plan.json`

### B. 文档计划结构

建议包含这些字段：

- `mode`: `project_docs` | `fallback`
- `reasoningSources`: 参与判断的规则文件列表
- `formalDocsRoot`: 正式文档根目录
- `formalDocTargets`: 正式文档类型到目标路径的映射
- `artifactPolicy`: 过程产物写入策略
- `confidence`: 模型自评把握
- `fallbackReason`: 触发回退时的原因

`formalDocTargets` 只针对正式文档，例如：

- `prd`
- `story`
- `trd`
- `api_design`
- `task`
- `feature_plan`

### C. 产物分类

系统在写文件前先做分类，而不是先写出去再事后补救。

#### 正式文档

符合以下条件的内容才允许进入项目正式文档区：

- 会在后续开发中被继续引用
- 是项目知识库的一部分
- 对应 PRD、Story、TRD、API 设计、Task、长期方案等正式产物

#### 过程产物

符合以下任一条件即视为过程产物：

- 仅服务当前会话
- 是阶段性的分析或证据
- 是中间稿、对照稿、检查稿、摘要稿
- 没有明确作为长期项目文档被引用

过程产物路径统一为：

- `.magpie/sessions/<capability>/<session-id>/...`

### D. 正式文档写入规则

当模型给出的正式文档落点满足以下条件时，允许写入项目正式文档区：

- 落点在仓库允许的文档目录内
- 不指向 `docs/templates/`
- 不退化为把过程文档平铺写到 `docs/` 根目录
- 路径与当前项目已存在的文档结构不明显冲突

满足条件：

- 正式文档写入模型规划出的目标位置

不满足条件：

- 不写入项目正式文档区
- 退回保守默认位置

### E. 保守默认位置

当模型无法可靠判断，或系统判定路径不合理时，正式文档进入保守默认位置：

- `.magpie/project-docs/<session-id>/`

这样做的目标是：

- 保住产物
- 不污染项目正式 `docs/`
- 允许用户后续手动确认或迁移

### F. 恢复与复用

会话恢复时优先复用既有 `document-plan.json`。

只有在以下情况才重新生成文档计划：

- 计划指向的目录已经失效
- 用户明确要求调整项目文档规则
- 现有计划被校验判定为无效

## 对现有能力的影响

### `loop`

- 新增文档计划生成
- 各阶段写文件前先走分类与路由
- `prd_review`、`domain_partition`、阶段报告等默认视为过程产物
- 正式文档型阶段结果才允许进入项目文档目录

### `harness`

- 在进入 `loop` 前生成或加载文档计划
- 会话级别保存 `document-plan.json`
- `status` / `inspect` 可展示当前文档模式与正式文档落点

### `.magpie`

新增一类更清晰的保守默认目录：

- `.magpie/project-docs/<session-id>/`

用于存放“应当是正式文档，但本轮无法可靠判断项目落点”的文档候选。

## 失败与回退策略

- 模型判断清楚且路径通过校验：正式文档进入项目 `docs/`
- 模型判断不清楚：正式文档进入 `.magpie/project-docs/<session-id>/`
- 模型判断出的路径明显不合理：拒绝采用，回退到 `.magpie/project-docs/<session-id>/`
- 过程产物试图进入正式文档区：系统强制拦截，写回 `.magpie/sessions/...`

## 验收标准

- 新会话不再把阶段过程文档平铺到项目 `docs/` 根目录
- 正式文档落点优先依据项目 `AGENTS.md` 与文档规则生成
- 同一会话中的正式文档落点保持一致
- 模型无法可靠判断时，正式文档回退到 `.magpie/project-docs/<session-id>/`
- 会话恢复后继续复用既有文档计划
- `status` / `inspect` 能看出本轮使用的是项目文档模式还是保守回退

## 第一版范围

- 先落文档计划、产物分类、正式文档路由、保守回退
- 先让正式文档“写对地方”，过程产物“不进正式区”
- 暂不处理旧文档迁移
- 暂不把项目审批状态机强绑定进 `harness`

## 风险

- 模型读取项目规则后仍可能给出“看起来合理但其实不符合团队习惯”的路径
- 不同项目的 `AGENTS.md` 粒度不同，第一版必须接受“部分项目会落到保守默认位置”
- 如果正式文档类型判断过宽，过程文档仍可能被误判为正式文档，因此分类边界必须由系统先兜住

## 推荐落地顺序

1. 增加 `document-plan.json` 生成与持久化
2. 增加产物分类层，先把过程产物完全隔离到 `.magpie`
3. 增加正式文档路径校验与保守回退
4. 将 `status` / `inspect` 接入文档模式展示
5. 为不同项目规则补测试样例
