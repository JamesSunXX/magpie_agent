# Magpie PRD 评审流程

## 目标

在进入开发前，用统一流程评审产品需求文档（PRD），确保需求完整、边界清晰、领域职责明确，并且可被实现、测试和上线。

## 适用范围

- 新功能
- 现有能力的显著改造
- 影响多个能力域（`review`、`discuss`、`trd`、`quality`）的需求

## 角色与职责

| 角色 | 主要职责 |
| --- | --- |
| PM（需求 owner） | 提交 PRD、说明业务目标与验收标准、闭环修改 |
| Tech Lead（评审 owner） | 主持评审、形成结论、确认技术边界 |
| Domain Owner | 评审领域职责、服务边界、依赖与风险 |
| QA | 评审可测性、验收口径、回归范围 |
| Reviewer（可选） | 安全、性能、可维护性专项评审 |

## 输入与输出

### 输入

- PRD 主文档
- 相关背景资料（历史需求、线上问题、竞品分析）

### 输出

- 评审结论：`通过` / `有条件通过` / `驳回`
- Action Items 清单（负责人、截止时间、状态）
- 更新后的 PRD 版本
- 需求追踪矩阵（需求 -> 领域 -> 技术方案 -> 测试用例）

## 评审流程

### 1. 发起评审（T-2 工作日）

PM 发起评审并提交以下材料：

- PRD 链接与版本号
- 变更摘要（本次相较上版改了什么）
- 待决策问题列表

### 2. 预审（异步）

Tech Lead 和 Domain Owner 在会前完成结构化预审，检查：

- 目标是否可量化
- 范围/非范围是否明确
- 关键流程是否完整
- 数据、接口、依赖是否可追踪

预审发现的问题进入 Action Items，会议优先讨论高风险项。

### 3. 正式评审会（同步）

建议 45-60 分钟，按以下顺序推进：

1. PM 讲解需求背景与目标（5-10 分钟）
2. Domain Owner 逐项确认领域职责划分（10-15 分钟）
3. Tech Lead 评估技术可行性与复杂度（10-15 分钟）
4. QA 确认可测性、验收标准、回归策略（10 分钟）
5. 收敛结论与待办（5-10 分钟）

### 4. 结论判定

结论采用以下规则：

- `通过`：无阻塞问题，可进入领域服务技术需求文档（TRD）阶段
- `有条件通过`：存在中低风险问题，限定时间内补齐后进入 TRD
- `驳回`：存在高风险或关键缺失，修订后需重新评审

### 5. 会后闭环（T+1 工作日）

Tech Lead 输出评审纪要，PM 在约定时间内更新 PRD 并回填 Action Items 状态：

- `open`：待处理
- `in_progress`：处理中
- `done`：已完成

当所有阻塞项为 `done` 后，方可进入下一阶段。

## 评审清单（Checklist）

### 业务与范围

- 目标是否清晰且可衡量
- 用户价值与优先级是否明确
- 范围/非范围是否可判定

### 领域职责划分

- 需求是否映射到明确领域
- 领域间边界是否清楚
- 跨领域依赖是否可追踪

### 技术可行性

- 关键实现路径是否存在明显阻塞
- 非功能需求（性能、安全、稳定性）是否覆盖
- 风险与降级/回滚策略是否定义

### 测试与发布准备

- 验收标准是否可执行
- 单测、mock、集成测试范围是否明确
- 发布准入条件是否定义

## 与开发链路的衔接

PRD 评审通过后，按以下链路推进：

`产品需求文档 -> 产品需求文档评审 -> 领域职责划分 -> 领域服务技术需求文档 -> 代码开发 -> 单元测试/mock 测试 -> 集成测试 -> MR & 上线`

本流程对应第二步“产品需求文档评审”，并为第三步和第四步提供输入质量保障。
