# Magpie 自监控与自修复设计

## 目标

让 `magpie` 在任务失败后，先稳住现场，再判断是否需要修复自身；当确实存在系统性缺陷时，允许它走一条独立的“修自己”支线，修完验证通过后，再恢复原任务继续执行。

本方案解决 4 个问题：

1. 原任务失败后，如何可靠暂停、记录现场、等待恢复
2. 如何区分临时故障、环境问题、实现问题、流程设计问题
3. 什么时候应该重试或降级，什么时候应该修 `magpie` 自己
4. 如何让“修自己”可控，不把系统越改越坏

## 适用范围

本方案覆盖：

- `loop`
- `harness`
- `harness-server`
- 当前仓库 `.magpie/sessions/` 下的会话持久化
- 仓库内自修任务的创建、验证和恢复规则

本方案暂不覆盖：

- 跨仓库调度
- 自动创建远程合并请求
- 自动部署或自动发布
- 无限制的连续自我演化

## 成功标准

以下条件同时满足，才算第一版完成：

1. 原任务失败后，不会丢失可恢复状态
2. 系统能给出结构化失败归因，而不是只留原始日志
3. 可自动处理的失败会被自动处理，不需要立刻进入“修自己”
4. 只有重复出现、证据明确的系统性问题，才会触发自修支线
5. 自修支线失败时，不会污染原任务状态
6. 自修完成后，只有验证通过，原任务才会恢复
7. 任意一次自修都存在明确停手机制和回退边界

## 当前基线

仓库里已经有一部分基础能力：

- `loop` 已支持测试失败后的有限次数修复与执行重试
- `loop` 已支持 `paused_for_human`、可靠点和部分恢复语义
- `harness` 已支持阶段化执行、评审回合、失败落盘和恢复继续
- `harness-server` 已支持排队、重试、重新入队和中断恢复
- `knowledge` 已支持沉淀 `failure-pattern`

但当前仍有明显缺口：

- 失败分类规则分散在不同流程里，没有统一判断层
- 大多数失败只影响当前会话，没有稳定升级为“系统缺陷”
- 系统不知道“这次该重试、该降级、该诊断，还是该修自己”
- 缺少独立的自修支线，会导致“原任务”和“修自己”混在一起

## 核心原则

1. 先恢复，再自修  
   原任务先停稳、保存现场、可恢复，再决定是否修 `magpie` 自己。

2. 原任务与自修任务隔离  
   自修必须作为独立任务运行，不能直接在失败现场边改边跑。

3. 先判断故障类型，再决定动作  
   不能把所有失败都当成代码问题。

4. 自修必须经过完整闭环  
   至少经过 `discuss -> plan -> implement -> review -> verify`，通过后才允许恢复原任务。

5. 自修必须可熔断  
   如果同类失败越来越多，或者修复后没有改善，系统必须停止自动自修。

## 总体架构

整个流程分成 4 层：

### 一、失败观测层

负责把失败统一记录成结构化事件。

每个失败事件至少包含：

- `sessionId`
- `capability`
- `stage`
- `timestamp`
- `errorSignature`
- `failureCategory`
- `failureReason`
- `isRetryable`
- `isSelfHealCandidate`
- `evidencePaths`
- `lastReliablePoint`

建议新增统一失败分类：

- `transient`
  - 超时
  - 限流
  - 断连
  - 临时 provider 故障
- `environment`
  - 命令不存在
  - 目录不正确
  - 权限问题
  - 配置缺失
- `quality`
  - 实现未通过测试
  - 结果不满足验收要求
- `prompt_or_parse`
  - 输出格式不稳定
  - 提示词导致阶段判断失真
  - 结果解析失败
- `workflow_defect`
  - 状态流转错误
  - 恢复语义错误
  - 阶段设计导致循环
- `unknown`

### 二、恢复决策层

负责根据失败分类决定下一步动作。

第一版动作固定为 5 类：

- `retry_same_step`
- `retry_with_backoff`
- `run_diagnostics`
- `degrade_path`
- `spawn_self_repair`

推荐的基础决策规则：

- `transient`  
  优先重试，必要时退避；超过阈值再停
- `environment`  
  先跑诊断，不进入自修
- `quality`  
  先做当前任务内修复，超过阈值后再考虑是否是系统问题
- `prompt_or_parse`  
  优先重跑解析、切换更稳的提示或更保守的流程
- `workflow_defect`  
  直接作为自修候选
- `unknown`  
  先进入诊断，不直接改自己

### 三、自修支线层

只有在满足触发条件时，系统才会创建独立自修任务。

自修支线的标准流程：

1. 冻结原任务并写入恢复点
2. 基于失败证据创建自修任务
3. 用 `magpie discuss` 明确问题与方案
4. 产出计划
5. 实施修复
6. 做代码评审和额外检查
7. 运行验证
8. 只有验证通过，才标记“可恢复原任务”

这条支线必须带上原任务的上下文摘要，而不是直接重放全部日志。

### 四、恢复执行层

自修通过后，不是直接把原任务标记成功，而是恢复到最后可靠点继续跑。

恢复规则：

- 如果原任务停在同一步骤内，优先从该步骤重跑
- 如果原任务停在半完成状态，只允许退回最后可靠点
- 如果恢复后再次命中同一失败签名，立刻触发熔断，不重复自修

## 自修触发条件

只有同时满足以下条件，才允许触发自修支线：

1. 失败不属于明显的临时故障
2. 已有足够证据，能说明问题更像 `magpie` 自身缺陷
3. 当前失败签名在近期重复出现，或单次影响足够严重
4. 原任务状态已保存，可安全恢复
5. 当前仓库工作区可进入独立修复流程

满足以下任一情况时，应优先判定为自修候选：

- 同一阶段连续多次命中 `prompt_or_parse`
- 同一失败签名在不同任务里重复出现
- 恢复逻辑本身出错
- `harness-server` 反复把任务重新入队却无法收敛
- 多个模型在评审阶段都指向同一流程缺陷

以下情况禁止直接自修：

- 单次超时
- 单次限流
- 本地命令没装
- 用户工作区本身测试就是坏的
- 缺少足够证据，不能判断问题位置

## 最小恢复底座

这是自修能力之前必须先具备的基础能力。

### 一、统一失败账本

每次失败都落一条标准记录，并能按失败签名聚合。

第一版建议位置：

- 会话内：`<session>/failures/*.json`
- 仓库级：`.magpie/failure-index.json`

### 二、稳定的恢复点

当前可靠点语义需要扩展到 `harness` 和 `harness-server`，不只停留在 `loop`。

第一版建议统一可靠点语义：

- `queued`
- `claimed`
- `planning_completed`
- `development_completed`
- `review_completed`
- `verification_completed`
- `resume_ready`

### 三、诊断步骤

对于 `environment` 和 `unknown`，先跑标准诊断，不直接改代码。

第一版诊断应至少回答：

- 当前命令能否执行
- 当前配置是否可读
- provider 是否可用
- 关键产物文件是否存在
- 当前状态是否与会话落盘一致

### 四、熔断规则

必须有明确的自动停止条件。

第一版建议：

- 同一失败签名最多触发 1 次自修
- 同一原任务最多进入 1 次自动自修支线
- 自修后恢复原任务，如果同一签名再次出现，则停止自动处理
- 仓库级失败率在短窗口内明显上升时，暂停自动自修

## 自修任务设计

自修任务不是普通失败重试，而是专门修 `magpie` 的工作流。

第一版建议新增一种内部任务类型：

- `workflow_self_repair`

它至少要包含：

- `sourceSessionId`
- `sourceCapability`
- `sourceFailureSignature`
- `problemSummary`
- `evidencePaths`
- `expectedFix`
- `resumeTarget`

这类任务的输入应由失败账本自动生成，不由调用方临时拼接。

## 推荐工作流

当原任务失败后，推荐执行顺序如下：

1. 记录失败事件
2. 保存原任务恢复点
3. 跑恢复决策
4. 如果只是可恢复故障，直接自动处理
5. 如果命中自修条件，创建独立自修任务
6. 自修任务走 `discuss -> plan -> implement -> review -> verify`
7. 自修通过后，把原任务标记为 `resume_ready`
8. 从最后可靠点恢复原任务
9. 恢复成功后，更新失败账本与长期知识

## 分阶段落地路线

### Phase 1：打底

目标：先让所有失败可观测、可归类、可恢复。

- 建统一失败事件结构
- 给 `loop`、`harness`、`harness-server` 接入统一失败记录
- 建失败签名聚合
- 扩展可靠点和恢复状态
- 加基础诊断

### Phase 2：自动恢复

目标：让常见故障不需要进入自修。

- 接入恢复决策层
- 支持退避重试
- 支持诊断后降级
- 让常见 `prompt_or_parse` 问题优先通过更保守路径兜住

### Phase 3：独立自修支线

目标：针对系统性缺陷，能独立修 `magpie` 自己。

- 新增自修任务类型
- 从失败账本自动生成自修输入
- 接入 `discuss -> plan -> implement -> review -> verify`
- 自修成功后标记原任务 `resume_ready`

### Phase 4：仓库级学习与熔断

目标：防止系统反复修同一类问题。

- 失败模式升级到长期知识
- 同类失败重复时直接命中已有规则
- 加仓库级熔断和人工接管提示

## 主要风险

### 一、误把环境问题当成系统缺陷

后果：白白修改 `magpie`，还可能引入新问题。

应对：

- 环境类失败先诊断
- 禁止单次环境故障直接进入自修

### 二、自修任务和原任务相互污染

后果：状态混乱，恢复点失效。

应对：

- 原任务与自修任务分 session
- 原任务在自修完成前保持冻结

### 三、无限自修循环

后果：不断创建修复任务，系统无法收敛。

应对：

- 同签名只修一次
- 同原任务只自动自修一次
- 恢复后同签名复发即熔断

### 四、修复通过但实际没解决问题

后果：恢复原任务后再次失败。

应对：

- 自修必须带验证
- 恢复原任务时优先跑最小复现场景

## 推荐的第一版范围

第一版不要追求“完全自治”，而要先做到“安全地开始自治”。

建议第一版只做：

1. 统一失败事件
2. 统一失败分类
3. 统一恢复策略
4. 原任务冻结与恢复点
5. 只针对 `workflow_defect` 和重复 `prompt_or_parse` 触发自修
6. 自修完成后恢复原任务

这样可以尽快把闭环搭起来，同时把改坏的风险压在可控范围内。

## 决策结论

`magpie` 可以自己改自己，但必须按“原任务冻结、独立自修、验证通过、再恢复原任务”的模式执行。

重点不是禁止自修，而是给自修加上足够结实的安全带。只有这样，多模型讨论、规划、实施、评审和合并这条链路，才会真正变成一个可靠的自我修复系统，而不是一个会放大错误的自动改写器。
