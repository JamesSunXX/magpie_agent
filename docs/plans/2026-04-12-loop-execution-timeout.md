# Loop Execution Timeout

## 背景

`loop` 默认把执行模型的超时写死成 15 分钟。对短任务够用，但在 `domain_partition`、`code_development` 这类阶段里，大任务经常因为统一时长过短而被提前截停。之前只能靠环境变量临时放大，既不稳定，也不方便在不同仓库里长期复用。

## 目标

- 把执行超时从硬编码改成配置优先
- 支持按任务复杂度自动放大
- 支持按阶段单独覆盖
- 保留最小值和最大值，避免估得过短或无限拉长
- `init` 和 `upgrade` 默认补齐这组配置

## 方案

在 `capabilities.loop.execution_timeout` 下新增 4 类配置：

- `default_ms`
- `min_ms`
- `max_ms`
- `complexity_multiplier`
- `stage_overrides_ms`

每次进入新阶段时，先取该阶段的基础值：

- 有阶段覆盖时，用阶段覆盖值
- 没有阶段覆盖时，用全局默认值

再乘以任务复杂度对应的倍数，并把结果限制在最小值和最大值之间。算出的结果会在阶段进入时写入事件流，方便排查现场。

## 默认值

- 默认超时：15 分钟
- 最小超时：5 分钟
- 最大超时：60 分钟
- 复杂度倍数：
  - `simple`: 1
  - `standard`: 2
  - `complex`: 3

## 影响

- 新生成配置会直接带上这组默认值
- 升级旧配置时会自动补齐这组默认值
- `loop` 阶段通知和事件会带出这次实际采用的时长
- 具体仓库可以在不改代码的情况下单独调大 `domain_partition`、`code_development` 等阶段

## 验证

- 增加了配置生成与升级测试，确保默认值会被写入
- 增加了 `loop` 测试，确保阶段会按“阶段覆盖 + 复杂度倍数”计算时长，并把实际值写进事件记录
