# 多模型工程协作流引擎进展盘点（截至 2026-04-13）

## 结论

`Magpie` 已经不是概念验证阶段。

到目前为止，仓库里已经有一套可运行的“单需求多模型闭环”主线：需求输入后，可以进入讨论、约束整理、开发执行、评审回合、测试验证、失败后继续修复，以及后台托管和中断恢复。

但它还没有发展成“多任务图谱式工程协作中枢”。也就是说，当前更准确的定位是：

- 已成型：单个需求的多模型协作闭环
- 已启动但未成型：图谱产物、依赖感知调度基础
- 未成型：正式图谱入口、完整图谱级工作台

## 已完成

### 1. 统一入口已经成立

当前仓库已经把以下能力收拢到同一个本地 CLI 入口：

- `review`：多 AI 代码评审
- `discuss`：多模型讨论
- `trd`：PRD 转 TRD，并落可机读约束
- `loop`：目标驱动的单任务闭环执行
- `harness`：需求到交付的闭环入口
- `harness-server`：后台托管 `harness` 队列
- `tui`：任务工作台

这说明“多模型工程协作流”作为一个产品入口已经存在，不再是零散脚本或临时流程。

### 2. 单任务质量闭环已经完成

围绕单个需求，当前主线已经具备这些稳定能力：

- `trd` 产出仓库级约束文件
- `loop` 在开发前读取约束并执行质量卡点
- 对适合的小任务先确认失败测试，再继续实现
- 测试失败后区分“继续修”与“执行事故”
- 超过阈值后暂停到人工介入，而不是无限重试
- 会话产物和状态持续写回仓库 `.magpie/`

这一阶段对应的核心目标，可以认为已经从“想法”走到“可用第一版”。

### 3. 长任务后台托管已经落地

`harness-server` 已经把 `harness` 从“一次命令跑到底”推进到“可后台运行、可恢复、可查看”的形态。当前已经具备：

- 后台服务启动、停止、状态查看
- 任务提交、列表、接回输出、恢复续跑
- 队列持久化
- 中断任务重新入队
- 服务重启后的会话恢复

这意味着长期运行能力已经有了基础实现，不再只是前台串行执行。

### 4. 多模型回合与检查面板已经成型

当前 `loop` 和 `harness` 已经共享一套较稳定的回合产物和查看方式：

- 每轮会保留参与者、评审结论、裁定结果、开放问题和下一步
- `status`、`inspect`、`attach` 可以直接查看持久化结果
- TUI 已经能展示会话摘要和补充说明
- 会话恢复时会尽量从可靠检查点继续，而不是从头猜状态

这部分说明“协作流”的回合记录、检查和恢复语义已经具备骨架。

### 5. 近期收尾能力在持续补强

截至 2026-04-13，最近一批提交集中在把现有主线补得更稳、更顺手，包括：

- `loop` 自动生成更像样的语义分支名
- `loop` 成功结束后可选自动创建 GitLab MR
- MR 创建失败时不误伤开发成功结论，但会明确记录需要人工补做
- `harness` 在已知模型错误场景下做降级兜底
- 裁定结果解析与恢复路径继续补强

这说明当前研发重心仍然是把现有单任务闭环做稳，而不是马上跳到下一代编排系统。

### 6. 图谱审批与工作台已经补出第一版可用面

截至 2026-04-13 的最新进展，图谱这条线已经不只是“底层有状态”：

- `harness` 已经有显式 `approve` / `reject` 命令
- 可以对整张图或指定节点写入批准 / 驳回结果
- 批准结果会持久化，重启后不会丢
- `status`、`inspect`、`list` 已经能直接看图谱汇总和节点细节
- TUI 选中图谱会话时，已经能看到待批准项、阻塞原因、当前可运行节点，以及建议优先批准的目标和命令

这意味着“图谱级审批完全缺失”这个判断已经过期。更准确的说法是：**审批和图谱提示已经有第一版可用面，但完整图谱入口和真正的全景工作台仍未完成。**

## 进行中

### 1. 单任务闭环的稳定性和收尾体验仍在补

虽然主线已经成型，但最近的真实工作重点仍然是“把第一版跑稳”。目前仍在持续打磨的方向包括：

- 自动提交、自动分支、自动提 MR 这类收尾动作
- 已知模型失败场景下的自动换路和降级
- 恢复语义、状态对齐、落盘一致性
- 文档规划模式、知识摘要、通知等配套体验

因此，这条线当前更像“稳定化阶段”，而不是“已完全封板”。

### 2. 大需求编排的下一阶段已经进入实现，但主入口还没成型

今天已经有了明确的下一阶段目标：把单任务闭环升级为“能拆大任务、按依赖推进、支持安全并行、带全景工作台”的系统。

这部分已经不再只是方案。随着 `sch/2026-04-13-milestone-4` 合入 `main`，仓库里已经补进了图谱执行的第一层基础能力：

- 图谱产物结构与落盘
- 节点依赖校验
- 冲突范围避让
- 图确认 / 节点审批等待状态
- 后台服务根据图谱判断“现在是否有可运行节点”

因此，这一阶段现在更准确的说法是：**已经开始实现，而且审批 CLI 与图谱提示工作台已经补出第一版，但离“做完 Milestone 4”还有明显距离。**

## 未开始或尚未落地

### 1. 大任务拆分与任务图谱

当前系统已经支持图谱产物本身，但还没有把“大需求自动拆成图谱”正式接到用户入口。现在主线上仍然看不到一个面向用户的“提交图谱任务”流程，主入口依旧主要是“一个需求 -> 一条闭环会话”。

### 2. 依赖感知的并行调度

当前后台能力已经开始接入图谱感知，已经能根据节点依赖、冲突范围和审批状态判断某个图谱会话是否具备可运行节点。

但它还没有做到完整的图谱级交付体验，比如：

- 只有前置节点完成后才启动后续节点
- 互相冲突的节点不并行
- 真正独立的节点安全并行
- 以图谱级状态解释“为什么能跑 / 为什么还在等”

### 3. 显式人工审批关卡的第一版已落地，但还不算做完

现在图谱基础层已经不只是“显示等待审批”：

- CLI 已经能直接批准或驳回图谱 / 节点
- 审批决定已经会写回会话文件
- 工作台已经能提示哪一项在等批准

但这还不是完整审批系统。当前仍然缺：

- 更完整的审批历史回看与筛选
- 从工作台内直接执行批准动作
- 更系统化的审批策略与批量处理

### 4. 全景工作台已有图谱提示层，但还没有完整总览

当前 TUI 已经不只是看单会话摘要。图谱会话被选中时，已经能看到：

- 哪个节点在等批准
- 哪个节点被什么原因卡住
- 当前有没有可继续推进的节点
- 建议优先批准哪一项

但它还没有形成真正面向多节点协作的总览工作台。也就是说，暂时仍然看不到：

- 一个大需求下所有子任务的整体进度
- 哪些节点被谁卡住
- 哪些节点可以下一步启动
- 哪些节点在等审批

## 里程碑判断

如果把这条线按阶段看，可以做出下面这个判断：

| 阶段 | 状态 | 说明 |
| --- | --- | --- |
| 单任务质量闭环内核 | 已完成 | `trd + loop` 主线已经成型 |
| 角色编排与检查视图 | 已完成 | 回合产物、查看和恢复语义已落地 |
| 后台托管基线 | 已完成 | `harness-server` 已能托管、恢复、续跑 |
| 单任务主线稳定化 | 进行中 | 最近提交仍在补强收尾和兜底 |
| 多任务图谱执行基础 | 已开始实现 | 图谱产物、依赖校验、冲突避让、审批等待状态已合入 `main` |
| 多任务图谱正式入口 | 未开始完成 | 还没有对外完整暴露图谱提交、查看和操作入口 |
| 图谱级审批 | 第一版已落地 | CLI 已可批准 / 驳回图谱或节点，结果可持久化并影响图谱状态 |
| 图谱级工作台 | 第一版已落地 | 已能提示待批准项、阻塞原因和建议优先批准目标，但还不是完整总览工作台 |

## 当前准确定位

截至 2026-04-13，`Magpie` 最准确的描述不是“完整的多模型工程协作流引擎”，而是：

> 一个已经可运行的单需求多模型闭环执行系统，外加后台托管、恢复和检查能力；  
> 同时图谱执行已经补出审批命令和工作台提示的第一版可用面，但还没把这条线做成完整可直接使用的多任务协作中枢。
