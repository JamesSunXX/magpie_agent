# Milestone 2：架构卡点与 TDD 闭环实施规格

## 目标

在已经完成仓库内会话落盘的基础上，把 Magpie 的下一步能力收敛成一套“单任务质量闭环内核”：

- `trd` 不只生成给人看的文档，还要生成机器可读约束
- `loop` 在真正改代码前，先检查是否违反约束
- `loop` 在写业务代码前，先写失败测试，再进入修复闭环
- 本地测试失败后，系统能提取错误证据并自动回传给执行阶段继续修

这一步的重点不是“让任务后台跑更久”，而是“让单个任务的工程质量闭环先成立”。

## 为什么现在做

Milestone 1 已经把会话状态和落盘目录统一到仓库内 `.magpie/sessions/`，解决了“状态住哪儿”的问题。  
Milestone 2 要解决的是“状态拿来怎么约束行为”和“代码改动怎么形成可验证闭环”。

当前仓库已经有：

- `magpie trd` 生成 TRD 文档
- `magpie loop` 执行多阶段流程并持久化会话
- `harness` 具备开发、评审、测试、修复的基础链路

当前仍缺的关键能力：

- TRD 没有稳定的机器可读约束出口
- 执行前没有统一的约束校验卡点
- 测试先行还不是默认强制路径
- 本地测试失败后，错误信息没有统一沉淀成“下一次修复输入”
- “执行事故重试”和“质量不达标继续修”还没有清晰区分

## 范围

本规格只覆盖“单任务质量闭环内核”：

- `magpie trd` 生成 `.magpie/constraints.json`
- `loop` 在写代码前执行约束校验
- `loop` 在单任务内执行 TDD 钩子与修复重试
- 本地测试命令执行、失败证据提取、重试阈值控制
- 这些结果如何落盘，供未来其他能力复用

本规格不覆盖：

- `harness-server`
- 后台常驻、任务队列、优先级、仓库互斥、租约锁
- 服务重启恢复
- 多任务并发调度
- 固定的 `claw -> kiro -> 仲裁` 双检查后台链路

## 与 Harness 长期运行方案的边界

这份规格和“Harness 长期运行闭环补齐方案”存在共享底座，但职责必须分开。

### 本规格负责

- 定义“单个任务如何在质量上跑完”
- 定义约束卡点、测试先行、失败修复、完成判定
- 定义单轮结果、失败证据、未解决清单等产物格式

### 长期运行方案负责

- 定义“任务如何在后台长期运行”
- 定义任务队列、并发、重投、恢复、attach/list/status
- 定义服务级状态机和调度策略

### 共享但不重复实现的部分

- 失败证据结构
- 未解决清单结构
- 单轮结果摘要格式
- 最后可靠点（last reliable point）语义

后续 `harness-server` 应直接复用这套单任务质量闭环，而不是另写一套测试先行和修复逻辑。

## 核心结果

Milestone 2 完成后，应能稳定做到下面这件事：

给定一个边界清楚的小需求，例如数学计算、字符串转换、数据映射或 BFF 层纯转换逻辑，Magpie 能：

1. 从 PRD/TRD 提取可执行约束
2. 先生成失败测试
3. 证明测试当前确实失败
4. 再生成最小实现
5. 自动运行本地测试
6. 若失败，提取错误证据并继续修复
7. 在阈值内把测试跑绿，或明确停在“需要人工介入”的状态

## 功能需求

### R1. TRD 约束产物

`magpie trd` 除了现有 TRD 文档外，还必须生成仓库内 `.magpie/constraints.json`。

这个文件至少要包含：

- `version`
- `sourceTrdPath`
- `generatedAt`
- `rules`

每条 `rule` 至少包含：

- `id`
- `category`
- `description`
- `severity`
- `scope`
- `checkType`
- `expected`
- `forbidden`

第一版重点支持的规则类型：

- 依赖限制
  - 例如：禁止引入 `axios`
- 路径规范
  - 例如：BFF 接口统一放在指定目录
- API 约束
  - 例如：必须走 `/api/v2/*`
- 测试要求
  - 例如：新增转换逻辑必须包含对应 `.test.ts`

第一版不要求把所有 TRD 内容都结构化。只需要把那些“能在执行前明确卡住”的约束抽出来。

### R2. 执行前约束卡点

`loop` 在进入代码改动前，必须先装载约束并做校验。

校验分两层：

1. 计划层校验
   - 当前任务目标、改动范围、目标文件是否明显触碰禁止区
2. 结果层校验
   - 生成的测试文件、实现文件、依赖改动、路径改动是否违反约束

约束校验结果必须分为三类：

- `pass`
- `needs_revision`
- `blocked`

语义要求：

- `pass`：允许继续
- `needs_revision`：允许重新生成测试/实现方案，但不能继续当前结果
- `blocked`：明确停下，等待人工处理

### R3. TDD 钩子

当任务类型属于“可通过本地测试验证”的需求时，`loop` 必须先走测试先行。

第一版适用范围：

- 新增纯函数
- 数据转换逻辑
- 工具函数
- 不依赖复杂外部环境的服务层逻辑

第一版不强制覆盖：

- 需要完整浏览器环境的前端交互
- 依赖真实外部服务的集成任务
- 涉及大规模迁移的复杂改造

TDD 钩子最少包含 4 个子步骤：

1. 生成测试目标说明
2. 生成测试文件
3. 运行目标测试，确认当前为失败态
4. 只有在“确认失败”后，才允许进入实现阶段

如果测试一开始就通过，不能直接视为成功，必须进入以下分支之一：

- 判断为已有行为已满足需求，结束任务
- 判断测试无效或覆盖不足，回到重新写测试

### R4. 本地测试执行与失败证据提取

实现阶段提交后，系统必须自动执行本地测试命令。

第一版支持两种来源：

- 从项目已有约定推断
- 从 TRD / 约束中显式给出

执行结果必须沉淀为结构化结果，而不是只保留原始终端输出。

最少输出字段：

- `command`
- `startedAt`
- `finishedAt`
- `exitCode`
- `status`
- `failedTests`
- `stderrSnippet`
- `stdoutSnippet`
- `evidencePath`

失败证据要服务下一轮修复，所以必须额外提炼：

- 失败测试名
- 首个关键报错
- 相关文件路径
- 是否更像测试问题、实现问题，还是环境问题

### R5. 修复重试策略

要明确区分两类问题：

1. 质量未达标
   - 测试失败
   - 约束校验未通过
   - 这类问题进入“继续修”

2. 执行事故
   - 命令超时
   - CLI 断连
   - 测试命令本身不可执行
   - 这类问题进入“事故重试或人工介入”

第一版建议默认阈值：

- 同一修复回合最大 3 次
- 同一执行事故原地重试最大 2 次

状态语义：

- `revising`
- `retrying_execution`
- `blocked_for_human`
- `completed`

达到阈值后必须留下明确原因，不能只写“失败”。

### R6. 单任务状态与产物

Milestone 2 必须把“单任务质量闭环”需要的状态和产物固定下来，供未来 `harness-server` 直接复用。

建议在现有 `.magpie/sessions/loop/<sessionId>/` 下新增或强化以下产物：

- `constraints.snapshot.json`
- `tdd/target.md`
- `tdd/red-test-result.json`
- `tdd/green-test-result.json`
- `repairs/attempt-*.json`
- `repairs/open-issues.md`
- `repairs/evidence/*.txt`

`session.json` 至少要能表达：

- 当前是否已通过约束检查
- 当前是否已确认红灯测试
- 当前修复次数
- 当前执行事故次数
- 最后可靠点
- 最后失败原因
- 当前未解决清单路径

这里的“最后可靠点”第一版建议限制在这些节点：

- `constraints_validated`
- `red_test_confirmed`
- `implementation_generated`
- `test_result_recorded`
- `completed`

恢复时只允许从最后一个完整节点继续，不从半截输出猜测状态。

## 目标流程

单任务质量闭环在第一版固定为以下顺序：

1. 读取 TRD 与 `.magpie/constraints.json`
2. 生成当前任务的约束快照
3. 做计划层约束校验
4. 生成测试目标说明
5. 生成测试文件
6. 运行目标测试，确认红灯
7. 生成实现代码
8. 做结果层约束校验
9. 运行本地测试
10. 若失败，提取证据并形成未解决清单
11. 在阈值内继续修复
12. 通过后写入最终结果

第一版不把 `claw` / `kiro` 固定检查器串进来。  
那属于长期运行的 `harness` 外层编排，不属于这里的单任务质量内核。

## 命令与能力影响

### `magpie trd`

新增能力：

- 生成 `.magpie/constraints.json`
- 在 TRD 会话里记录约束产物路径

第一版不要求：

- 对已有 TRD 自动回填历史约束
- 做很重的约束迁移

### `magpie loop`

新增能力：

- 在适用任务上强制插入 TDD 钩子
- 在改代码前和产出后分别做约束校验
- 运行本地测试并结构化沉淀结果
- 失败时生成修复输入并驱动下一次尝试

### 其他能力

`harness`、`issue-fix`、后台服务后续都可以复用这套内核，但不要求在 Milestone 2 里全部打通。

第一优先级是：

- `trd`
- `loop`
- 可独立验证的最小任务样例

## 验收标准

### 必须满足

1. `magpie trd` 运行后，仓库内出现可读的 `.magpie/constraints.json`
2. `loop` 在适用任务上，先生成测试文件，再生成实现文件
3. 系统能证明测试在实现前处于失败态
4. 系统能自动运行本地测试，并把失败结果结构化落盘
5. 测试失败时，系统能基于错误证据发起下一轮修复
6. 超过阈值后，任务进入明确的人工介入状态，而不是假成功
7. 这些状态和产物能在会话目录中稳定找到

### 示例验收任务

- 输入一个明确的纯函数需求
  - 例如金额格式化、字段映射、字符串规范化
- 系统自动生成失败测试
- 系统生成实现
- 系统自动把测试跑绿
- 中间若失败，能在落盘结果里看到修复证据和重试记录

## 测试计划

### 单元测试

- 约束结构解析与校验
- 规则命中与阻断分支
- TDD 阶段状态转换
- 测试执行结果解析
- 修复阈值控制
- 最后可靠点推进

### 集成测试

- `trd` 生成 TRD + constraints
- `loop` 在简单需求上走完整红灯 -> 绿灯流程
- 约束冲突任务在执行前被卡住
- 测试命令超时时进入执行事故分支

### 回归测试

- 不适合 TDD 的任务不会被错误强制进入测试先行
- 已有会话状态不会因为新增字段而无法读取
- 现有 `loop inspect`/`resume` 仍可读取关键状态

```bash

npm run test:coverage
npm run build
npm run check:docs
```

## 实施拆分建议

建议按 4 个小步落地：

### Step 1. 约束产物

- 为 `trd` 增加 `.magpie/constraints.json`
- 先支持最小一组高价值规则

### Step 2. 约束卡点

- 在 `loop` 里接入执行前和结果后两层校验
- 明确 `pass / needs_revision / blocked`

### Step 3. TDD 内核

- 先覆盖纯函数、转换逻辑这类简单任务
- 打通红灯确认、实现生成、测试执行

### Step 4. 修复与证据

- 统一失败证据格式
- 接入修复重试与人工介入状态

## 风险与控制

### 风险 1：把所有任务都硬塞进 TDD

控制：

- 第一版只覆盖可本地验证的简单任务
- 明确不适用类型，避免假闭环

### 风险 2：约束规则过重，导致几乎所有任务都被卡死

控制：

- 第一版只支持高确定性规则
- 对模糊规则只给 `needs_revision`，不直接 `blocked`

### 风险 3：把“执行事故”和“质量未达标”混成一类

控制：

- 两类状态、计数器、产物分开记录
- 后续后台调度层直接复用这两个分支语义

### 风险 4：和 Harness 长期运行方案重复建设

控制：

- 本规格只定义单任务质量闭环
- 不在本轮引入后台宿主、队列、并发、服务恢复
- 共享字段和产物命名提前固定，避免后续二次迁移
