# Harness Server Execution Loop

## 背景

`harness` 原本更像一次性命令：启动后一直占着当前终端，命令超时或终端断开时，任务也容易跟着中断。对需要多轮修复、评审和验收的长任务来说，这个模式不够稳。

## 这次落地的范围

### 1. 后台托管入口

- 新增 `magpie harness-server start|status|stop`
- 后台状态持久化到当前仓库 `.magpie/harness-server/state.json`
- 支持前台运行和 tmux 托管两种启动方式

### 2. 提交改为“直跑或入队”

- 当后台服务未运行时，`magpie harness submit` 继续按原样直接执行
- 当后台服务已运行时，`magpie harness submit` 改为只创建会话并入队
- `magpie harness attach|status|inspect|list` 继续复用原有会话体系

### 3. 队列与恢复

- 队列会记录输入、重试次数、最近错误和最后可靠位置
- 增加轻量优先级：`interactive`、`high`、`normal`、`background`
- 可识别超时、断连、限流等可重试错误，并转入等待重试状态
- 服务重启后，会把半途中的会话重新排回队列
- 已完成的开发阶段会优先接着原来的开发会话继续
- 已完成的评审轮次会从 `rounds.json` 恢复，不会从第一轮重来

### 4. 每轮双检查

- `harness` 每轮在原有评审和单测判断后，固定补跑 `claw` 检查
- 同一轮再固定补跑 `kiro` 检查
- 未解决问题会并入同一轮结论，用于后续修复和是否结束的判断

## 本轮不做的事

- 不做跨仓库统一调度
- 不做单仓库多任务并发执行；当前仍保持串行，先保证工作区安全
- 不做中途“精确续跑到某一步”；恢复后先显式标成阻塞，避免错误续跑

## 验收标准

- 后台服务启动后，`harness submit` 不再占住当前终端
- 已入队任务会被后台服务取走执行
- 高优先任务先于低优先任务执行
- 每轮都会产出 `claw` 和 `kiro` 的检查结果文件
- 常见临时错误会进入等待重试，而不是立刻算最终失败
- 服务重启后，未完成任务会从上一个已保存节点继续推进
- 文档入口能说明新的用法和落点
