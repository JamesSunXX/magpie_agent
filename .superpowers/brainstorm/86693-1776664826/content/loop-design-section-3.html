Loop 新方案 v1: 配置和异常处理

这一页解决最后两件事：正式阶段怎么配置工具，以及异常轮次怎么处理但又不把配置搞炸。

配置原则
按正式阶段配置

每个正式阶段都可以配置：

primary: 主说 / 主执行

reviewer: 复核 / 挑战

rescue: 异常接管 / 返工接管

异常先继承

像“超时接管”“开发返工”“验证返工”这种异常轮次，先继承所属正式阶段的配置。

这样先把主线跑顺，不额外增加一套异常配置表。

建议的配置长相
示意
loop:
  stages:
    prd_review:
      primary: claude-code
      reviewer: gemini-cli
      rescue: kiro-cli
    implementation:
      primary: codex
      reviewer: gemini-cli
      rescue: kiro-cli
    integration_test:
      primary: codex
      reviewer: gemini-cli
      rescue: kiro-cli

意思很简单：只在正式阶段上写谁主说、谁复核、谁救援。

异常处理规则

超时或执行事故

先交给当前阶段的 rescue 接管。

如果 rescue 也没法完成，再由阶段裁定决定是返工还是停住。

开发返工

不跳回一个模糊的大开发阶段。

而是回到 green_fixup 或 implementation 继续，仍沿用该阶段配置。

验证返工

失败时标成“验证返工”或“联调返工”。

允许补修，但阶段身份不变，避免把问题冲淡成普通开发。

验证和落地检查

文档侧

  • 阶段定义文档要更新成 9 段
  • 每段交付物字段要固定
  • 阶段配置说明要补到能力文档里

实现侧

  • 现有阶段枚举要改
  • 现有 planner / executor 绑定要改成按阶段读取
  • 现有 fallback 和确认逻辑要改成按阶段继承配置
Y

这段也对

设计已经可以收敛成正式文档。

A

这段还要调

配置方式或异常规则还要改。
