Loop 当前结构草图

先把“阶段”和“工具角色”分开看。这个图描述的是我从现有代码里读出来的真实现状，不是建议方案。

当前阶段

1. prd_review

先看需求，形成阶段结论。

2. domain_partition

拆任务和边界，产出执行计划。

3. trd_generation

整理可执行设计和约束。

4. code_development

真正改代码。这里还混着约束检查、红测确认、修复重试。

5. unit_mock_test

跑项目定义的单测 / mock 检查。

6. integration_test

跑联调或更高层验证。

当前工具角色
Planner 角色

默认：claude-code

负责计划、阶段评估、风险判断、是否继续重试。

如果配置覆盖，也可以改成别的工具。

Executor 角色

默认：codex

负责执行当前阶段的实际工作，尤其是开发和修复。

如果 codex 超时，当前实现会自动切到 kiro 继续。

当前判断链
阶段完成后的判断顺序

Planner 先判断阶段结果。

如果只是低把握或普通失败，可进入多模型确认。

多模型确认会调用 reviewers；如果意见不一致，再交给 summarizer 裁定。

只有高风险或模型明确要求人工，才转人工确认。
